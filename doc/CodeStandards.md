# Coding Standards

This document defines the required C# coding standards for this library. These rules apply to all contributor-authored source code. Every code contribution generated by an AI Agent must strictly adhere to these patterns.

The project targets Unity 2019.4 and uses BepInEx with Harmony for runtime patching. Do NOT import or utilize any namespaces or modern .NET syntax incompatible with the Unity 2019.4 Mono runtime.

A Roslyn analyzer is in development to better enforce these rules. Until it is available, contributors and AI agents are responsible for applying them manually during implementation and code review.

---

## General Principles

Code must be readable, structurally consistent, and intentional.

Do not add whitespace solely to vertically align declarations, assignments, arguments, operators, comments, or chained member access.

```cs
// Correct
private readonly string _name;
private readonly int _count;

// Incorrect
private readonly string _name;
private readonly int    _count;
```
Do not use whitespace as a substitute for structural scope. Control-flow statements must either contain their complete body on the same line or use braces.
```cs
// Correct
if (isEnabled) return;

if (isEnabled)
{
    Initialize();
}

// Incorrect
if (isEnabled)
    Initialize();
```
Generated code is excluded from these formatting requirements. Decompiled code is not generated code for the purpose of this policy and must be reformatted before it is committed.

---

## Indentation and Braces

Use four spaces per indentation level (configure your IDE settings as such). Do not use tabs. 

Use Allman brace style for all declarations and multi-line blocks. Open and close braces MUST be on their own lines.
```cs
public sealed class Example
{
    public void Execute()
    {
        if (_isEnabled)
        {
            Run();
        }
    }
}
```
Single-line control-flow statements are allowed only when the entire statement (including the body) fits completely on a single line.
```cs
if (value == null) return;
if (!isValid) throw new ArgumentException(nameof(isValid));
```
else, catch, and finally must always use braces, even when their body contains only one statement.
```cs
try
{
    Load();
}
catch (Exception exception)
{
    ModInterface.Log.Error(exception);
}
finally
{
    Cleanup();
}

// Incorrect
catch (Exception exception)
    ModInterface.Log.Error(exception);
```
The same brace and scope rules apply to if, else, for, foreach, while, do, switch, using, lock, try, catch, and finally.

---

## Line Wrapping and Method Chains

Do not add indentation to align punctuation, parameters, operators, assignments, or member-access operators.

Long member-access chains must be split across separate lines. Each continuation line is indented exactly once relative to the initial expression. Do not align . characters vertically.
```cs
var result = source
    .Where(item => item.IsEnabled)
    .Select(item => item.Name)
    .ToList();

// Incorrect
var result = source
                  .Where(item => item.IsEnabled)
                  .Select(item => item.Name)
                  .ToList();
```
Use the same principle when wrapping long method calls, constructors, collection initializers, and argument lists: indent continuation lines once, never to create visual alignment.

## Naming

### Types

Class, struct, record, interface, enum, delegate, and public nested-type names must use PascalCase.
```cs
public sealed class ModController
{
}
```
Interfaces must begin with 'I'.
```cs
public interface IAssetProvider
{
}
```
Generic type parameters must use PascalCase and begin with T.
```cs
public sealed class Cache<TValue>
{
}
```
### Fields (Private Marker Convention)

Private fields use specific prefix/suffix rules depending on how they are accessed. The character x is literal text.

| Scenario | Rule | Example |
| :--- | :--- | :--- |
| Standard Private Field | _camelCase | private int _currentIndex; |
| Accessed EXCLUSIVELY via Getter | Prefix x_ | private readonly int x_currentIndex; |
| Accessed EXCLUSIVELY via Setter | Suffix _x | private string _currentIndex_x; |
| Accessed via Both Getter & Setter | Both x_ and _x | private string x_currentIndex_x; |

Do not bypass the intended accessor or marker convention from within the declaring class.
```cs
public int CurrentIndex => x_currentIndex;

// Correct
public void MyMethodA(int value)
{
    if (value == CurrentIndex)
    {
        //do work
    }
}

// Incorrect
public void MyMethodB(int value)
{
    if (value == x_currentIndex)
    {
        //do work
    }
}
```
### Variables and Parameters

Local variables and method parameters must use camelCase.
```cs
public void LoadAsset(string assetName)
{
    var loadedAsset = FindAsset(assetName);
}
```
ALWAYS use var whenever possible, even when the resulting type is obscured (e.g., returning from a method call). Explicit types must ONLY be used when strictly necessary for compilation, such as when a variable is declared before it is initialized.
```cs
// Correct
var controller = new ModController();
var names = new List<string>();
var result = GetComplexType();

// Incorrect
ModController controller = new ModController();
ComplexType result = GetComplexType();
```
### Constants and Pseudo-Constants

Constants and static readonly fields that act as pseudo-constants must use ALL_CAPS_SNAKE_CASE.
```cs
public const string MOD_GUID = "example.huniepop2.mod";
private static readonly TimeSpan DEFAULT_LOAD_TIMEOUT = TimeSpan.FromSeconds(10);
```
### Properties, Methods, Events, and Enums

Properties, methods, events, enum types, and enum values must use PascalCase.
```cs
public string DisplayName { get; }
public event Action Loaded;

public enum LoadState
{
    NotStarted,
    Loading,
    Complete
}
```
Enums are discouraged. Do not introduce an enum when the value domain may expand outside the values known at the time of implementation. These will typically be replaced with a set of RelativeId instances. Enums shall never be used as instance identifiers. If you think you need an enum, you should probably be making a class instead.

## Access Modifiers

Every type member must explicitly declare an access modifier.

Use internal by default. Use public only when a member is intended to be part of the supported library API.
```cs
internal sealed class AssetLoader
{
    internal void Load()
    {
    }
}
```
Because this project is primarily an API library, public accessors are expected where they are necessary to expose stable, documented functionality.

---

## Properties and Backing Fields

Avoid auto-properties on non-trivial classes.

When a class implements behavior beyond simple data storage, use explicit private backing fields and explicit properties.
```cs
public sealed class AssetReference
{
    private string _assetName;

    public string AssetName
    {
        get => _assetName;
        set => _assetName = value;
    }
}
```
Auto-properties are acceptable for simple data-transfer objects, immutable configuration objects, or other intentionally trivial types.

Expression-bodied members are encouraged when they remain clear.

public bool IsLoaded => _asset != null;

public override string ToString() => Name;

---

## Member Ordering

Members must appear in the following order from top to bottom:

1. Constants.
2. Public nested types.
3. Public properties, with their private backing fields directly beneath each property.
4. Private and protected fields.
5. Constructors.
6. Initialization methods.
7. Cleanup methods.
8. Other methods, organized into topical groups.
9. Event-handler methods.

Public and private methods must be grouped by topic rather than separated solely by access level.
```cs
public sealed class AssetManager
{
    public const string DEFAULT_ASSET_PATH = "assets";

    public sealed class AssetLoadResult
    {
    }

    public string AssetPath
    {
        get => _assetPath;
        set => _assetPath = value;
    }

    private string _assetPath;

    private readonly Dictionary<string, object> _assets;

    public AssetManager()
    {
        _assets = new Dictionary<string, object>();
    }

    public void Initialize()
    {
    }

    public void Cleanup()
    {
    }

    public object GetAsset(string assetName)
    {
        return FindAsset(assetName);
    }

    private object FindAsset(string assetName)
    {
        return _assets[assetName];
    }

    private void OnSceneLoaded()
    {
    }
}
```

## Namespaces and File Organization

Use file-scoped namespaces.
```cs
namespace HuniePop2Modding;
```
A source file may contain multiple top-level types when doing so improves cohesion, such as a primary type with closely related internal helper types.

Do not split tightly coupled small types into separate files merely to enforce a one-type-per-file convention.

---

## Documentation and Comments

All public API must have XML documentation comments.
```cs
/// <summary>
/// Loads a game asset by its registered identifier.
/// </summary>
/// <param name="assetId">The identifier of the asset to load.</param>
/// <returns>The loaded asset.</returns>
/// <exception cref="KeyNotFoundException">
/// Thrown when no asset is registered with <paramref name="assetId" />.
/// </exception>
public object LoadAsset(string assetId)
{
    return _assets[assetId];
}
```
Comments must explain intent, rationale, game-specific behavior, compatibility constraints, or non-obvious implementation details. Do not write comments that merely restate the code.

Do not commit any of the following to production branches:

* TODO comments.
* FIXME comments.
* Commented-out code.
* Temporary debugging code.
* Placeholder implementations.

Use issues, pull requests, or project tracking instead of leaving unfinished work in source comments.

## Exceptions and Logging

Use ModInterface.Log for project logging.

Load-time operations must allow exceptions to propagate unless there is a documented recovery path. This ensures mod-loading failures are visible and actionable. These are considered similar to a compilation error and should be immediately visible when launching the game.
```cs
public void LoadDefinitions()
{
    _definitions = ReadDefinitions();
}
```
Gameplay operations must handle expected runtime failures, log sufficient diagnostic context, and preserve gameplay stability where possible.
```cs
public void ApplyGameplayEffect()
{
    try
    {
        ApplyEffectInternal();
    }
    catch (Exception exception)
    {
        ModInterface.Log.Error($"Failed to apply gameplay effect: {exception}");
    }
}
```
Log messages must identify the failed operation and include relevant identifiers or state when available. Do not silently swallow exceptions.

## Unity, BepInEx, and Harmony

Keep Unity lifecycle methods and Harmony patch methods small. Move substantial logic into named methods that can be tested, reviewed, and reused.

Avoid unnecessary allocations in per-frame, UI-refresh, and frequently invoked gameplay code. In particular, NEVER allocate collections, strings, delegates, closures, and LINQ query results inside Update, LateUpdate, repeated UI refresh methods, and high-frequency Harmony patches unless the allocation is demonstrably acceptable.

Harmony patch classes must have names that clearly identify the patched target and their purpose.
```cs
[HarmonyPatch(typeof(UiStatusPortrait))]
internal static class UiStatusPortraitPopulatePatch
{
    [HarmonyPostfix]
    private static void Postfix(UiStatusPortrait __instance)
    {
        RefreshPortrait(__instance);
    }

    private static void RefreshPortrait(UiStatusPortrait portrait)
    {
    }
}
```
Prefer narrowly targeted patches. Do not use broad patches, reflection scans, or runtime discovery when a stable direct target is available.

## RelativeId

A RelativeId is a class instance used in place of traditional integer-based IDs to support a modular, multi-plugin ecosystem. 

Because this library coordinates multiple interacting plugins and mods simultaneously, relying on raw integers for asset, data, or object identification introduces severe ID collision risks. A RelativeId resolves this by establishing isolated namespaces for each plugin.

### Composition

A RelativeId consists of two core components:
1. **SourceId:** A unique identifier assigned to the source by the base mod.
2. **LocalId:** The specific identifier given to the asset or object within its own source namespace.

Together, they form a globally unique identifier across the base game and all loaded plugins, preventing registration conflicts and allowing separate teams or developers to define their own namespaces independently.

### Usage and Constraints

* **Enum Substitution:** Enums shall never be used as instance identifiers. When an object or asset domain requires expansion outside of the values known at compilation time, a set of stable RelativeId instances must be used instead of an enum.
* **Lookup Maps:** When tracking persistent game data or state within an expansion class, always use the RelativeId as the lookup key rather than relying on unstable runtime references or integer keys.
* **Serialization:** RelativeId instances must be serialized consistently to preserve their namespace boundaries across save states and network boundaries.

## Expansion Files and Expanded Types

Use the expansion pattern when the library needs to attach additional state, behavior, lifecycle handling, or mod-facing API to an existing game type without modifying the base-game class directly.

An expansion file may contain the following closely related top-level types:

1. An extension class that exposes the expansion from the base-game type.
2. A Harmony patch class that forwards relevant base-game lifecycle or behavior hooks.
3. An Expanded<TypeName> class that stores mod-specific state and implements the expanded behavior.

These types are intentionally colocated because they form one cohesive integration unit. Do not separate them into unrelated files merely to enforce a one-type-per-file convention.
```cs
namespace Hp2BaseMod;

public static class GirlDefinition_Ext
{
    /// <summary>
    /// Gets the mod expansion associated with this girl definition.
    /// </summary>
    public static ExpandedGirlDefinition Expansion(this GirlDefinition definition)
        => ExpandedGirlDefinition.Get(definition);
}

[HarmonyPatch(typeof(UiAppDisplaySlot))]
internal static class UiAppDisplaySlotPatch
{
    [HarmonyPatch("Start")]
    [HarmonyPostfix]
    public static void Start(UiAppDisplaySlot __instance)
        => ExpandedUiAppDisplaySlot.Get(__instance).Start();
}

internal sealed class ExpandedUiAppDisplaySlot
{
}
```
### Naming

Extension classes for expanded game types must use the base type name followed by _Ext.
```cs
public static class GirlDefinition_Ext
{
}
```
Expansion classes must use the Expanded prefix followed by the complete base type name.
```cs
public sealed class ExpandedGirlDefinition
{
}
```
Harmony patch classes must use the base type name followed by Patch.
```cs
internal static class UiAppDisplaySlotPatch
{
}
```
When multiple patches target the same base type, use an additional PascalCase suffix that identifies the patched behavior.
```cs
internal static class UiAppDisplaySlotTooltipPatch
{
}

internal static class UiAppDisplaySlotLifecyclePatch
{
}
```
### Expansion Lookup and Instance Tracking

Every expansion type must expose a static Get method that returns the single expansion associated with a core instance.
```cs
public static ExpandedGirlDefinition Get(GirlDefinition definition)
{
    return Get(definition.id);
}
```
Use a private static lookup collection to associate each core object or stable game-data identifier with its expansion.

Use a stable identifier such as RelativeId when the expanded object represents persistent game data. Use the core object reference when the expanded object represents a runtime Unity object or UI instance.

private static Dictionary<RelativeId, ExpandedGirlDefinition> _expansions = new();

private static Dictionary<UiAppDisplaySlot, ExpandedUiAppDisplaySlot> _expansions = new();

The Get method must create an expansion only when one does not already exist.
```cs
public static ExpandedUiAppDisplaySlot Get(UiAppDisplaySlot core)
{
    if (!_expansions.TryGetValue(core, out var expansion))
    {
        expansion = new ExpandedUiAppDisplaySlot(core);
        _expansions[core] = expansion;
    }

    return expansion;
}
```
Do not create multiple expansions for the same core instance or identifier.

### Core Instance References

An expanded type must retain a reference to its base-game object when it needs to interact with that object after construction.
```cs
protected UiAppDisplaySlot _core;

private ExpandedUiAppDisplaySlot(UiAppDisplaySlot core)
{
    _core = core;
}
```
Constructors for expansion types should normally be private or internal. Expansion instances must be created through the static Get method so the instance-tracking invariant is preserved.

### Harmony Patch Responsibilities

Harmony patches must remain thin forwarding layers. A patch should identify the target method and delegate meaningful work to the expansion type.
```cs
[HarmonyPatch("OnDestroy")]
[HarmonyPostfix]
public static void OnDestroy(UiAppDisplaySlot __instance)
    => ExpandedUiAppDisplaySlot.Get(__instance).OnDestroy();
```
Do not place substantial gameplay, UI, reflection, state-management, or event-subscription logic directly in a Harmony patch method. That logic belongs in the associated expanded type.

Use postfixes when the expansion must respond after base-game initialization or behavior completes. Use prefixes only when the expansion must alter, replace, or prevent the original behavior.

### Lifecycle and Cleanup

Expansion types that subscribe to events, retain runtime Unity object references, allocate disposable resources, or maintain runtime lookup entries must implement cleanup behavior.

The Harmony patch must forward the relevant base-game cleanup lifecycle method to the expansion.
```cs
[HarmonyPatch("OnDestroy")]
[HarmonyPostfix]
public static void OnDestroy(UiAppDisplaySlot __instance)
    => ExpandedUiAppDisplaySlot.Get(__instance).OnDestroy();
```
Cleanup methods must unsubscribe event handlers and release references that would otherwise retain destroyed Unity objects.
```cs
public void OnDestroy()
{
    ExpandedItemSlotBehavior.Get(_core.itemSlot).PreShowEvent -= OnTooltipPreShow;
}
```
When an expansion is keyed by a runtime object reference, its cleanup method must also remove the expansion from the static lookup collection.
```cs
public void OnDestroy()
{
    ExpandedItemSlotBehavior.Get(_core.itemSlot).PreShowEvent -= OnTooltipPreShow;
    _expansions.Remove(_core);
}
```
An expansion keyed by persistent game-data identifiers may remain registered for the lifetime of the loaded data set, but it must be cleared when that data set is unloaded, replaced, or reinitialized.

### Public Expansion APIs

Public extension methods and public members on expanded types are library API and require XML documentation comments.

Expansion classes should expose stable, intentional API rather than exposing internal dictionaries, base-game implementation details, or mutable state directly.

When mutable state must be exposed, prefer explicit methods or properties that preserve invariants over public mutable fields.

### Reflection in Expansions

Cache reflected members in static readonly fields when they are used repeatedly.
```cs
private static readonly MethodInfo ON_TOOLTIP_PRE_SHOW_METHOD
    = AccessTools.Method(typeof(UiAppDisplaySlot), "OnTooltipPreShow");
```
Reflection lookups must not be performed repeatedly in lifecycle callbacks, Harmony patches, UI refresh methods, or other high-frequency code paths.

If a required reflected member cannot be found during a load-time operation, allow the exception to propagate so the mod-loading failure is visible. If reflection fails during gameplay, log the failure with ModInterface.Log and preserve gameplay stability where possible.
